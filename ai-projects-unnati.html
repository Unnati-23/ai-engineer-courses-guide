Model Regression Detection System

A CI/CD-style pipeline that continuously tests any LLM-powered feature against a golden dataset whenever a prompt or model changes, detects quality regressions, and alerts your team via Slack before bad outputs reach users.

Every AI team ships prompt changes blind. This project proves you think about what happens after deployment — the exact mindset hiring managers are desperate for and almost no candidates demonstrate.

Tech Stack · 2026
Python 3.12 Claude API DeepEval RAGAS Langfuse GitHub Actions Slack Webhooks SQLite Docker Pydantic v2
01Define the LLM Feature Under TestDays 1–2
  • Build a customer support email classifier that returns a category + summary. Wrap in a single Python function with the prompt as a configurable parameter.
  • Version your prompts as YAML files in a /prompts directory — version ID, timestamp, system prompt, few-shot examples. This is the code you're running CI against.
  • Define a PromptConfig dataclass (Pydantic v2) with typed input/output contracts. Input: email text. Output: structured JSON.
02Build the Golden DatasetDays 2–4
  • Hand-curate 50–100 test cases. Do NOT generate with an LLM — these are human-verified ground truth. Include ambiguous, short, typo-filled, and sarcastic emails.
  • Tag each case with expected_difficulty (easy / medium / hard / adversarial). Store as versioned JSON with stable IDs.
03Build the Evaluation EngineDays 4–7
  • Write an async test runner using DeepEval and RAGAS. Score on: exact category match, summary faithfulness, latency, and token usage.
  • Build comparison logic: diff every run against the previous. Track pass rate delta, per-category delta, regressions (pass → fail), and improvements (fail → pass).
  • Add statistical thresholds: warn at 3% delta, critical at 8%. Make configurable. Trace every eval run in Langfuse.
04Alerting and Reporting LayerDays 7–9
  • Generate an HTML diff report: run metadata, scorecard vs baseline, regressed cases side-by-side, trend chart over last N runs.
  • Wire Slack webhooks: send pass/warn/fail status, headline numbers, and link to the full report. Add slow drift detection via 7-run moving average.
05Wire into CI/CDDays 9–11
  • GitHub Actions workflow: trigger on every PR touching /prompts. Run eval, post PR comment with pass/fail, block merge on critical regressions.
  • Containerize everything with Docker. Write the README as internal onboarding docs — not a tutorial.
06Polish for PortfolioDays 11–12
  • Record a 3-minute Loom: change a prompt → trigger eval → show Slack alert → walk through diff report.
  • Frame the write-up: "teams ship prompt changes blind — I built the CI/CD layer that catches regressions before users do."

LLM Cost Autopilot

An intelligent routing layer that sits in front of multiple LLM providers, analyzes each request's complexity, routes it to the cheapest model capable of handling it at acceptable quality, and continuously validates routing decisions.

Every company running LLMs at scale is bleeding money on over-provisioned model calls. This signals you understand AI engineering as a business problem — the gap between a junior and senior hire.

Tech Stack · 2026
Python 3.12 OpenAI API Anthropic API Ollama FastAPI scikit-learn Langfuse DuckDB 1.0 Grafana Docker
01Build the Unified Model InterfaceDays 1–3
  • Create a ModelConfig dataclass: provider, model ID, cost per input/output token, average latency, quality tier. Populate with real 2026 pricing for GPT-4o, Claude Sonnet, Claude Haiku, and local Llama via Ollama.
  • Build a single send_request(prompt, model_config) function returning a standardized Response with output, tokens, latency, cost, and model ID. Test all providers with the same 10 prompts.
02Build the Complexity ClassifierDays 3–6
  • Define 3 tiers: Tier 1 (extraction, basic Q&A) → cheapest model. Tier 2 (summarization, classification) → mid-tier. Tier 3 (multi-step reasoning) → highest quality.
  • Label 200+ prompts by hand. Train scikit-learn classifier (logistic regression or random forest). 80%+ accuracy on held-out set is V1-ready. Store routing map as YAML.
03Build the Async Quality Verification LoopDays 6–9
  • After returning the response, queue an async job: send the same prompt to the highest-tier model and score agreement. Divergence = routing failure. Log to Langfuse.
  • Auto-escalation: if verifier catches a failure, re-run on higher-tier model. Feed every routing failure back as a new classifier training example — the flywheel.
04Logging and Cost DashboardDays 9–11
  • Log every request to DuckDB: timestamp, prompt hash, complexity tier, routed model, cost, latency, quality score, escalation flag.
  • The money shot metric: display cost reduction % vs sending everything to GPT-4o. This number is the headline of your portfolio.
05Expose as an APIDays 11–13
  • Single POST /v1/completions endpoint — caller doesn't choose the model, the router does. Return response + metadata showing which model was selected and why.
  • Add GET /v1/stats (cost savings summary) and PUT /v1/routing-config (update mappings without redeploy). Docker-compose the full stack.
06Polish for PortfolioDays 13–14
  • Run 500–1,000 diverse prompts through the system. Generate cost savings report. Frame: "I built a system that reduced LLM API costs by X% while maintaining Y% quality parity."

Failure Forensics Tool for AI Pipelines

An observability layer for multi-step AI pipelines that traces every intermediate step, identifies exactly where failures originate when the final output is bad, and feeds flagged failures back into a growing evaluation dataset.

When a multi-step AI pipeline produces garbage, most teams have no idea which step broke. You're building a mini LangSmith — and being able to articulate why observability matters is a senior-level signal.

Tech Stack · 2026
Python 3.12 LangGraph v1.0 Claude API OpenTelemetry Langfuse Pydantic v2 PostgreSQL 16 Streamlit Docker
01Build the Multi-Step PipelineDays 1–3
  • Design a 4-step pipeline: Intake → Extraction → Classification → Summarization. Each step takes a Pydantic v2 typed input and returns a typed output. No spaghetti — clean isolated functions.
  • Deliberately inject failure modes: contracts with no dates, invoices in mixed currencies, ambiguous document types. You need failures to trace.
02Build the Tracing LayerDays 3–6
  • Every pipeline execution gets a unique trace_id. Each step becomes a Span capturing: input, output, LLM prompt, raw response, token count, latency, errors, and a self-confidence score (1–5) from the LLM.
  • Use OpenTelemetry + Langfuse. Store complete traces as JSON, index metadata in PostgreSQL. Decorator pattern so instrumenting a new step is one line of code.
03Build the Backward Trace AnalyzerDays 6–9
  • Walk backward through spans on failure. Use LLM-as-judge to score each step's output quality given its input. First significant quality drop = root cause.
  • Categorize failure types: Extraction Hallucination, Misclassification, Propagation Error, Prompt Failure, Context Loss. Build an evidence chain with specific input/output pairs.
04Build the Visual Trace ExplorerDays 9–11
  • Visualize pipeline as a node graph: green (healthy), yellow (low confidence), red (root cause). Clicking a node shows full span details.
  • Add a flagging button: mark any output as bad → system runs backward analysis → displays root cause with evidence.
05Build the Feedback-to-Eval LoopDays 11–13
  • Every confirmed failure auto-generates an eval test case: original input, failing step, bad output, human-corrected output, failure category. Append to growing dataset.
  • Build regression tracking: re-run accumulated cases, track "known issues resolved" trend over time.
06Polish for PortfolioDays 13–14
  • Frame: "I built an observability system that reduces MTTR for AI pipeline failures from hours of manual debugging to seconds of automated diagnosis." Record Loom showing the full diagnosis flow.

Self-Healing Technical Documentation

A GitHub Action that monitors a codebase, detects when code changes make documentation inaccurate, identifies the specific stale sections, and either auto-generates a PR with corrected docs or flags discrepancies for human review.

This project lives inside a CI/CD pipeline, not a Streamlit demo. It solves a universal engineering pain point and demonstrates the full AI engineering stack in a system other engineers would actually install.

Tech Stack · 2026
Python 3.12 text-embedding-3-small Pinecone Serverless Claude API Pydantic v2 PyGithub GitHub Actions Docker
01Build the Code-to-Docs MappingDays 1–4
  • Parse codebase into semantic chunks: function signatures, class definitions, API endpoints, config schemas. Each chunk gets a stable identifier (file path + name).
  • Build a link graph between doc sections and code chunks using heuristics + embedding similarity. Persist as a JSON index in the repo.
02Build the Change Detection PipelineDays 4–7
  • Parse git diff on every PR. Skip comment-only and whitespace changes. Focus on API signature changes, config changes, new/removed features.
  • For each meaningful change, query the link graph for affected doc sections — your "suspects." Verify staleness with Claude: old code + new code + doc section → is this still accurate?
03Build the Doc Repair EngineDays 7–10
  • For each confirmed stale section: send current doc + new code + staleness diagnosis → Claude rewrites only the stale parts, preserving original style and structure.
  • High confidence → auto-fix. Low confidence → draft with TODO markers + human review flag. A second LLM pass validates the correction before any PR is created.
04Build the GitHub ActionDays 10–12
  • Package as a GitHub Action with action.yml. Inputs: LLM API key, confidence threshold, auto-merge flag. Post a PR comment on every trigger: "Doc Check: 3 verified, 1 auto-fixed, 2 flagged."
05Test on a Real RepositoryDays 12–13
  • Fork FastAPI or Pydantic. Install your Action. Deliberately break docs via code changes. Measure: true positives, false positives, false negatives, correction quality.
06Polish for PortfolioDays 13–14
  • Publish to the GitHub Actions Marketplace — a published Action is a fundamentally different portfolio signal than a repo that just sits there.

LLM Output Arbitration System

A multi-agent pipeline that routes any LLM-generated output to multiple competing critic models that independently evaluate it for accuracy, consistency, and completeness, then synthesizes their critiques into a single confidence-scored verdict.

Instead of building a system that generates answers, you're building one that catches bad answers — this flips the script on every other portfolio project and demonstrates the evaluation mindset AI teams are actively hiring for.

Tech Stack · 2026
Python 3.12 LangGraph v1.0 Claude API OpenAI API Ollama Pydantic v2 instructor Langfuse FastAPI SQLite
01Design the Critic Agent ArchitectureDays 1–3
  • Three specialist critics: Factual Accuracy Critic (GPT-4o), Logical Consistency Critic (Claude), Completeness Critic (local Llama via Ollama). Different models = different blind spots = the most valuable signal.
  • Each critic returns a Pydantic v2 model via instructor: dimension, score (1–5), list of specific issues with quotes and severity, self-confidence rating.
02Build the Orchestration Layer with LangGraphDays 3–6
  • LangGraph state graph: input parsing → parallel critic dispatch → critique collection → disagreement detection → adjudication → verdict. Fan-out to all critics simultaneously.
  • Handle edge cases: one API fails → graceful degradation, continue with remaining critics. All critics agree it's perfect → short-circuit adjudicator.
03Build the Adjudicator AgentDays 6–8
  • Adjudicator receives all three critique reports + detected disagreements. Produces: overall quality score (1–10), confidence level, confirmed issues, dismissed flags with reasoning, one-paragraph summary.
04Build the Verdict Explorer UIDays 8–10
  • Inline annotations on the original text: red (confirmed issues), yellow (low-confidence flags), green (validated claims).
  • Side-by-side critic comparison panel. Agreements in green, disagreements in orange. This visual makes the architecture immediately comprehensible.
05API and AnalyticsDays 10–12
  • POST /v1/arbitrate + POST /v1/arbitrate/batch + GET /v1/arbitrations/{id}. Track which critic finds the most issues, which is overruled most often, most common failure types.
06Polish for PortfolioDays 12–14
  • Test with: planted factual errors, logically flawed arguments, responses that miss the point, and a genuinely good response. Frame: "AI models auditing each other's work."

Hybrid Search RAG over Internal Docs

A production-grade RAG system that ingests company documentation, indexes it with both dense vector and sparse keyword search, retrieves the most relevant context, and generates grounded answers with inline source citations and hallucination detection.

RAG is the single most requested skill in AI engineering job descriptions. Most candidates build a toy demo with a single PDF. You're building a system with hybrid retrieval, citation verification, and RAGAS evaluation — the concerns that separate real RAG engineers.

Tech Stack · 2026
Python 3.12 LangChain Claude API Pinecone Serverless rank_bm25 Cohere Rerank 4.0 RAGAS Langfuse FastAPI Docker
01Ingestion and Chunking PipelineDays 1–3
  • Accept markdown, text, HTML, PDF. Build three chunking strategies: fixed-size with overlap, recursive by section headers, semantic chunking via embedding similarity. Track which strategy each chunk used.
  • Embed every chunk with text-embedding-3-small. Store in Pinecone Serverless with metadata. Build BM25 index via rank_bm25 in parallel. Add near-duplicate deduplication (cosine > 0.95 = skip).
02Build the Hybrid Retrieval EngineDays 3–6
  • Dense retrieval (Pinecone) + sparse retrieval (BM25). Fuse with Reciprocal Rank Fusion (RRF) — configurable weighting (0.7 dense / 0.3 sparse).
  • Send top-20 candidates through Cohere Rerank 4.0 (released Dec 2025). Keep top 5. The reranking step is a strong interview talking point.
03Generation and Citation LayerDays 6–9
  • Prompt Claude to cite using bracketed references [1], [2] and to explicitly say when context is insufficient. Parse and verify every citation with LLM-as-judge. Flag unsupported claims.
  • Build a composite confidence score: retrieval relevance + citation coverage + answer completeness. Return alongside every answer.
04Evaluation FrameworkDays 9–11
  • Build a 50+ Q&A golden dataset by hand. Measure with RAGAS metrics: faithfulness, context recall, context precision, answer relevance. Run across all three chunking strategies. Data drives architecture decisions.
  • Trace all retrieval and generation steps in Langfuse. Every eval run is linked to observable traces.
05API and DashboardDays 11–13
  • POST /v1/ask returns answer + citations + confidence scores + source metadata. Build a Streamlit UI with clickable citations, retrieved chunk view, and a toggle comparing hybrid vs dense-only retrieval side-by-side.
06Polish for PortfolioDays 13–14
  • Frame: "I built a RAG system with hybrid search achieving X% faithfulness on a 50-question RAGAS eval suite." Lead with numbers. Show the chunking strategy comparison data.

Semantic Caching Layer for LLM APIs

A middleware caching service that sits between your application and any LLM provider, detects semantically similar requests already answered, and serves cached responses instantly — cutting latency to near-zero and reducing API costs by 30–60%.

Every company running LLMs at scale has the same problem: redundant API calls burning money. This proves you think about infrastructure efficiency — the kind of ROI engineering managers immediately understand.

Tech Stack · 2026
Python 3.12 FastAPI Redis Stack OpenAI Embeddings Claude API Langfuse Helicone Grafana Docker
01Build the Cache Index and Similarity EngineDays 1–3
  • Embed every incoming prompt with text-embedding-3-small. Store in Redis Stack (vector search built in). On each request, query for nearest neighbor — cosine similarity above threshold (start 0.95) = cache hit.
  • Cache key strategy: include system prompt hash + temperature + max_tokens + model. Two identical prompts with different system prompts must NOT share cache entries.
02Build the Drop-In Proxy APIDays 3–5
  • Mirror the OpenAI chat completions API contract exactly — any app switches by changing base URL only. Return same format with an added cache hit/miss header.
  • Streaming support: cache hits return instantly. Cache misses stream from provider to client while buffering for storage. Only cache complete, successful responses.
03Build Cache Policies and EvictionDays 5–8
  • TTL tiers: stable/factual queries (24h+), time-sensitive queries (1h or disabled). Build a classifier that auto-assigns TTL based on prompt content.
  • Similarity threshold tuner: expose endpoint showing the hit rate vs accuracy tradeoff at different thresholds. This is the core interview talking point. Implement adaptive thresholds by request type.
04Monitoring and AnalyticsDays 8–10
  • Track with Langfuse + Helicone: hit rate per model, latency (cached vs uncached), cost savings per hour, cache capacity, and similarity score distribution for near-misses.
  • Grafana dashboard: real-time hit rate, cumulative cost savings, P50/P95/P99 latency comparison. The cost savings chart is your portfolio headline visual.
05Containerize and Load TestDays 10–12
  • Docker-compose: FastAPI proxy + Redis Stack + Grafana + optional Ollama. Send 2,000+ requests, mix of unique and repeated queries. Measure hit rate convergence and total cost savings.
06Polish for PortfolioDays 12–14
  • Lead with: "A drop-in caching layer that reduced LLM API costs by X% and P95 latency by Y% on a 2,000-request load test." Record demo: same query — LLM hit → cache hit → semantically similar cache hit.

Text-to-SQL with Guardrails and Hallucination Detection

A natural language interface that translates plain English questions into SQL queries, executes them safely with guardrails preventing destructive operations, validates that the generated SQL actually answers the question asked, and presents results with a confidence score.

Text-to-SQL is one of the highest-value enterprise LLM applications. Building one with guardrails and hallucination detection proves you can ship AI features a compliance team would actually approve.

Tech Stack · 2026
Python 3.12 Claude API SQLAlchemy 2.x PostgreSQL 16 sqlglot Guardrails AI Pydantic v2 instructor FastAPI Docker
01Build the Schema-Aware Prompt EngineDays 1–3
  • Use SQLAlchemy 2.x to auto-introspect the database: tables, columns, types, primary/foreign keys, sample values for categorical columns. Build a dynamic prompt constructor that filters to relevant tables only (embedding similarity).
  • Handle ambiguity explicitly: when a question maps to multiple interpretations (e.g. "revenue" = gross or net), return a structured clarification request instead of guessing.
02SQL Generation and Safety LayerDays 3–6
  • Scan for prompt injection attacks first. Then generate SQL via instructor + Pydantic v2 structured output. Parse with sqlglot — block all DDL (CREATE, ALTER, DROP) and all DML writes (INSERT, UPDATE, DELETE).
  • Run generated SQL in a read-only PostgreSQL transaction with SELECT-only permissions. Validate output with Guardrails AI. Two-layer defence: middleware + DB permissions.
03Hallucination Detection SystemDays 6–9
  • Back-translate SQL → natural language. Compare against original question. Significant divergence = the SQL probably doesn't answer the right question. Score alignment, flag low confidence.
  • Multi-query validation: generate two independent SQL approaches, execute both. If results match → high confidence. If they diverge → flag and present both with explanations.
04Build the Query InterfaceDays 9–11
  • POST /v1/query returns: generated SQL, results, confidence score breakdown, and guardrail warnings. Streamlit UI with syntax-highlighted editable SQL, sortable results table, and query history.
05Build the Evaluation SuiteDays 11–13
  • 50+ golden question-SQL pairs. Measure: SQL exact match, execution match, hallucination detection rate, guardrail effectiveness. README leads with: "X% execution accuracy, zero unsafe queries."
06Polish for PortfolioDays 13–14
  • Lead with safety: "blocks 100% of destructive operations and detects Y% of hallucinated queries before they reach the user." Companies care more about not breaking things than accuracy percentages.

Prompt Versioning and A/B Testing Platform

A platform that treats prompts as versioned artifacts, lets teams deploy multiple prompt variants simultaneously, splits traffic between them, measures performance across custom metrics, and declares statistically significant winners.

Most AI teams change prompts by editing a string in production and hoping for the best. This project shows you understand prompt engineering at scale is an experimentation problem — the operational maturity of a senior AI engineer.

Tech Stack · 2026
Python 3.12 Claude API OpenAI API PostgreSQL 16 scipy.stats LangSmith FastAPI Pydantic v2 Streamlit Docker
01Build the Prompt RegistryDays 1–3
  • Every prompt gets: unique ID, version number, system prompt text, few-shot examples, model parameters, commit message, timestamp. Store in PostgreSQL. This is git for prompts.
  • Add rollback support: any version can be marked "active" — no code deploy needed. Log every activation with actor and reason. Diff endpoint showing what changed between any two versions.
02Build the Experiment EngineDays 3–6
  • Experiment schema: name, prompt ID, variant versions, traffic split %, primary metric, target sample size, status. Use consistent hashing on user ID so same user always sees same variant.
  • Auto-stop experiments on: variant error rate exceeding 10%, or losing variant performing significantly worse on primary metric. Notify experiment owner when triggered.
03Build the Metrics and Analysis LayerDays 6–9
  • Pluggable metric collectors. Built-in: latency, token usage, cost, error rate. Custom: LLM-as-judge quality score, task-specific accuracy. Run asynchronously after response.
  • Implement statistical significance testing: two-sample t-test, Mann-Whitney U, p-value, confidence interval. Track LangSmith for trace-linked metrics. Auto-declare winner at 95% confidence.
04Build the Management InterfaceDays 9–11
  • Dashboard: create experiments, configure splits, monitor live metrics with confidence intervals, promote winners to production with one click. Build a full audit log — who changed the prompt that broke the feature last Tuesday?
05Integration and TestingDays 11–13
  • Seed with a customer support classifier. Create three variants (zero-shot vs few-shot vs chain-of-thought). Run 500+ synthetic requests. Let it converge to a winner.
06Polish for PortfolioDays 13–14
  • Frame: "I built a prompt experimentation platform that brings feature-flagging rigor to LLM development — statistically significant results instead of guessing." Position as infrastructure.

LoRA Fine-Tuning Pipeline

An end-to-end pipeline that takes a domain-specific dataset, applies LoRA fine-tuning followed by DPO alignment to an open-source base model, evaluates the fine-tuned model against the base model on task-specific benchmarks, and packages the result for deployment with full experiment tracking.

Fine-tuning is where AI engineering meets ML engineering. Most candidates run a notebook once and call it done. Building a reproducible pipeline with DPO alignment and RAGAS evaluation proves you own the full model customization lifecycle.

Tech Stack · 2026
Python 3.12 PyTorch 2.x Hugging Face PEFT Unsloth BitsAndBytes TRL (DPO) W&B vLLM Llama 3.1 8B
01Build the Training DatasetDays 1–4
  • Choose a narrow, measurable domain: legal clause classification, medical note summarization, code review comments. Curate 500–2,000 instruction-format examples. 500 quality examples beat 5,000 noisy ones.
  • 80/10/10 train/val/test split. Test set is sacred — never train on it. Create 30–50 handcrafted edge-case examples for your benchmark.
02Set Up the Training PipelineDays 4–7
  • Load Llama 3.1 8B in 4-bit via Unsloth + BitsAndBytes — fits on a single RTX 4090. Unsloth provides 2× faster training than vanilla PEFT with 70% less VRAM.
  • Configure LoRA: r=16, α=16, all-linear target modules. Log all hyperparameters, loss curves, and validation metrics to W&B. Run hyperparameter sweep (rank 8/16/32 × LR 1e-4/2e-4/5e-4).
03Run DPO AlignmentDays 7–9
  • After SFT, apply Direct Preference Optimization (DPO) via TRL. Create preference pairs: chosen (better) vs rejected (worse) responses. DPO aligns the model to human preferences without a separate reward model.
  • Track both SFT and DPO stages in W&B. Every 2026 fine-tuning role expects the SFT → DPO pipeline, not SFT alone.
04Evaluate and Compare ModelsDays 9–11
  • Base model → benchmark first. Fine-tuned (best checkpoint) → same benchmark. Head-to-head on every example. Use LLM-as-judge (GPT-4o) for qualitative comparison.
  • Test for catastrophic forgetting: run standard benchmarks on both models. Document the tradeoffs if general capabilities dropped.
05Build the Inference PipelineDays 11–12
  • Export LoRA adapter weights separately (typically <100MB). Deploy via vLLM — load base model, attach adapter, expose chat completions API. Build A/B comparison endpoint: base vs fine-tuned side-by-side.
06Polish for PortfolioDays 13–14
  • Headline: "Fine-tuned Llama 3.1 8B on [domain] using LoRA + DPO, improved task performance from X% to Y% while maintaining Z% of general capabilities." Every number backed by evaluation data.

LLM Gateway with Rate Limiting, Fallback Routing, and Observability

A production API gateway that sits in front of all LLM calls, enforces per-team rate limits and budgets, automatically falls back to alternative providers on outage, and provides unified observability across every LLM interaction.

Every company with more than one team using LLMs ends up building something like this. It's pure infrastructure engineering applied to AI — lets you lead with your production engineering strengths.

Tech Stack · 2026
Python 3.12 FastAPI Redis 7 YAML + hot reload OpenTelemetry Langfuse Prometheus Grafana Docker
01Build the Unified Proxy LayerDays 1–3
  • Normalize requests across OpenAI, Anthropic, Ollama behind a unified interface. Caller never knows which provider served the request. Support streaming passthrough with simultaneous logging.
  • Request enrichment: inject standard system prompts, compliance disclaimers, or content filters per team. Centralizes policy without requiring every team to implement it.
02Rate Limiting and Budget EnforcementDays 3–6
  • Token bucket rate limiting via Redis 7: per-team requests/minute and tokens/minute. Atomic, distributed-safe. Return 429 with Retry-After headers.
  • Daily/monthly budget caps: compute cost per request in real-time. Warn at 80%, block at 100%. Admin API to adjust limits without restart. All changes logged with actor.
03Build the Fallback and Resilience LayerDays 6–9
  • Continuous health checking every 30s. Status: healthy/degraded/down per provider-model combination. Automatic fallback: OpenAI GPT-4o timing out → route to Anthropic Claude Sonnet.
  • Circuit breakers: N failures in M seconds → open circuit, route to fallbacks. After cooldown → half-open test request. Close circuit on success. Log every state change.
04Build the Observability LayerDays 9–11
  • OpenTelemetry spans for every stage. Langfuse for LLM-specific tracing. Prometheus metrics: RPS, error rate, latency P50/P95/P99, cost per team per day, fallback trigger rate.
  • Three Grafana dashboards: Operations (health, fallbacks), Business (spending, budgets), Performance (latency, throughput). Slack alerts for error rate spikes and budget thresholds.
05Load TestDays 11–13
  • 5,000+ concurrent requests. Measure gateway overhead latency (<10ms target), rate limiting accuracy under load, fallback behaviour under simulated outages.
06Polish for PortfolioDays 13–14
  • Frame: "I built an LLM API gateway handling X req/sec with <10ms overhead, automatic multi-provider failover, and per-team budget enforcement." Lead with reliability numbers.

AI Feature Flag System with Gradual Rollout and Quality Monitoring

A feature flag platform built for AI features that supports percentage-based rollouts, automatically monitors quality metrics during rollout, and triggers automatic rollback if output quality degrades below a configurable threshold.

Traditional feature flags are binary. AI features fail on a quality gradient. This project proves you understand that shipping AI features requires different operational patterns — and you can build the tooling to support it.

Tech Stack · 2026
Python 3.12 PostgreSQL 16 Redis 7 Claude-as-judge Langfuse FastAPI Slack Webhooks Pydantic v2 Docker
01Build the Flag Evaluation EngineDays 1–3
  • AI flag schema: rollout % (0–100), quality threshold, rollback trigger, baseline config, experimental config. Build a lightweight Python SDK — flag_client.evaluate(flag, context) — handling consistent user hashing and graceful degradation.
02Build the Quality Monitoring LayerDays 3–6
  • Async quality evaluator: after every flagged response, queue an eval job (Claude-as-judge scoring, latency check, error rate). Must not add latency to user-facing response.
  • Rolling quality windows: last 100 evals, last 1 hour, last 24 hours. Track mean, std dev, P10 quality. Compare experimental vs baseline continuously via Langfuse.
03Build Gradual Rollout AutomationDays 6–9
  • Staged rollout: 1% → 5% → 25% → 50% → 100%. At each boundary: quality above threshold → auto-advance. Quality dips → pause and alert. Statistical testing before each advance.
  • Shadow mode: run experimental variant on all traffic, log what would have been shown, evaluate offline. Catches catastrophic failures before any user is affected.
04Auto-Rollback TriggerDays 9–10
  • When experimental variant's P10 quality drops below threshold for 50+ consecutive evaluations: set rollout % to 0, fire Slack alert with quality data, log rollback with full context. Cooldown prevents flapping.
05Dashboard and Integration TestingDays 10–13
  • Dashboard: all flags status, real-time quality metrics, rollout schedule progress, one-click rollback. Build demo: flag starts at 0% → bad variant detected → auto-rollback with Slack alert → good variant reaches 100%.
06Polish for PortfolioDays 13–14

Automated Eval Dataset Generator from Production Logs

A system that continuously mines production LLM logs, identifies interesting, edge-case, and failure-mode interactions, and automatically converts them into labeled evaluation test cases — building an ever-growing, production-representative eval dataset without manual curation.

The hardest part of AI evaluation isn't the eval harness — it's the dataset. You're solving the data supply problem by turning production traffic into eval data automatically. This is a force multiplier AI teams dream about.

Tech Stack · 2026
Python 3.12 Claude API Pandas 2.x Pydantic v2 DuckDB 1.0 HDBSCAN RAGAS DeepEval LangSmith Celery
01Build the Log Ingestion LayerDays 1–3
  • Log schema: user prompt, system prompt, model, raw response, latency, tokens, user feedback signals (thumbs up/down, retries), feature name, timestamp. Store in DuckDB 1.0 for fast analytical queries.
  • Three sampling modes: uniform random, stratified by feature/model/error-status, and signal-boosted (oversample negative feedback, retries, high latency). Add configurable PII redaction.
02Build the Interaction ClassifierDays 3–6
  • Embed all sampled prompts, cluster with HDBSCAN. Each cluster = a natural category of questions your system handles. Outliers = your most valuable eval candidates.
  • Categorize quality with Claude-as-judge (1–5). High quality → positive test cases. Low quality → negative test cases. Build a difficulty estimator: simple / moderate / hard / adversarial.
03Build the Auto-Labeling PipelineDays 6–9
  • Generate golden answers with Claude (chain-of-thought). Create multi-dimensional labels: expected quality, expected behavior, key assertions must-contain, must-NOT-contain (hallucination traps), difficulty, category. Validate with Pydantic v2.
  • Confidence-based routing: high-confidence labels → auto-add to dataset. Low-confidence → human review queue. Deduplication: cosine similarity > 0.92 against existing cases = skip.
04Build the Eval RunnerDays 9–11
  • Score each response with RAGAS + DeepEval. Trace linked evaluations in LangSmith. Flag regressions: cases that used to pass but now fail. Dashboard shows dataset growth and coverage heatmap by cluster.
05Build the Curation DashboardDays 11–13
  • Streamlit UI: browse cases by category/difficulty, view full interaction, approve/edit/reject auto-labels. Set up Celery nightly cron: sample → cluster → label → deduplicate → append. Dataset grows automatically.

  • Frame: "I built a system that automatically converts production LLM logs into a growing eval dataset. In two weeks, it generated X test cases across Y categories with Z% auto-label accuracy — the dataset grows while you sleep."

Multi-Modal Document Processor with OCR, LLM Extraction, and Validation

An end-to-end pipeline that accepts any document format (PDF, image, scan), performs OCR to extract raw text, uses LLMs to extract structured data, and validates every extraction against configurable business rules — with human-in-the-loop review for low-confidence results.

Document processing is one of the largest enterprise AI deployment categories. It combines computer vision, NLU, and production engineering — every layer of the AI engineering stack solving a problem every large company is actively paying to solve.

Tech Stack · 2026
Python 3.12 Tesseract EasyOCR Claude Vision API instructor Pydantic v2 Celery Redis 7 PostgreSQL 16 Docker
01Build the Ingestion and OCR LayerDays 1–3
  • Accept PDFs, images, scans. For PDFs: try native text extraction first (PyMuPDF), fall back to OCR on low text density. Dual-engine OCR: Tesseract + EasyOCR — compare outputs, pick higher-confidence reading per segment.
  • Vision model fallback: for handwriting or complex layouts, send page image directly to Claude Vision API. More expensive but handles what traditional OCR cannot.
02Build the LLM Extraction EngineDays 3–6
  • Define Pydantic v2 schemas per document type: invoice (vendor, line items, totals, due date), contract (parties, effective date, obligations). Use instructor library to enforce schema in LLM output.
  • Chunk-and-merge for long documents: extract per chunk, merge results. Flag conflicts: two chunks extract different values for the same field → include both with source locations.
03Build the Validation and Business Rules EngineDays 6–9
  • Pydantic validators: dates in plausible ranges, amounts positive, required fields present, cross-field consistency (line items sum to invoice total).
  • Domain rules: vendor in known vendor list? Amount within expected range for this vendor? Statistical anomaly detection vs historical data for the same source.
04Build the Human Review InterfaceDays 9–11
  • Side-by-side: original document (image) on left, extracted data on right. Highlight source location per field. Color-coded confidence: green (high), yellow (medium), red (low/failed validation).
  • Inline editing with change logging: original value, corrected value, who corrected it, whether it was an extraction error or a false positive. This data feeds back to prompt refinement.
05Build Feedback Loops and AnalyticsDays 11–13
  • Every correction is a training signal: update few-shot examples, tune confidence thresholds, identify systematic extraction errors. Dashboard: auto-approval rate over time is the key efficiency metric.

Agent Orchestration System with Tool Use, Memory, and Human-in-the-Loop

A multi-agent orchestration platform where a supervisor agent decomposes complex tasks, delegates subtasks to specialized tool-using agents, maintains persistent memory across interactions, and escalates to a human operator when confidence is low or the task requires approval.

Most candidate projects are single-agent toy demos. Building a multi-agent system with real tool use, persistent memory, and human-in-the-loop escalation proves you can architect the kind of autonomous AI systems companies are actively building and hiring for right now.

Tech Stack · 2026
Python 3.12 LangGraph v1.0 Claude API MCP E2B Redis 7 PostgreSQL 16 Langfuse FastAPI Celery Docker
01Build the Agent ArchitectureDays 1–4
  • Three-layer hierarchy: Supervisor Agent (plans, delegates), Specialist Agents (research, analysis, writing, code execution), Reviewer Agent (validates before returning to supervisor). Each as a LangGraph v1.0 node with typed input/output schemas.
  • Register every tool via MCP (Model Context Protocol) — the 2026 standard for agent tool integration. Use E2B for sandboxed code execution. Rate limit per tool and log every invocation.
02Build the Memory SystemDays 4–7
  • Short-term working memory in Redis 7 — scoped to a single task, cleared on completion. Shared across all agents during execution.
  • Long-term semantic memory in PostgreSQL 16 — extract and store what worked, what tools were used, user preferences. Future tasks query memory before planning. This is how the system gets smarter over time.
03Build the Human-in-the-Loop SystemDays 7–10
  • Escalation triggers: low planning confidence, specialist fails twice, sensitive operations (financial transactions, external comms), reviewer score below threshold.
  • Approval levels: Notify → Approve action → Approve plan → Take over. Review UI shows context, proposed action, agent reasoning, and relevant past decisions. Chat panel to ask the agent clarifying questions.
04Build Observability and DebuggingDays 10–12
  • Every execution produces a trace tree in Langfuse: planning decisions, tool calls, reviewer evaluations, memory retrievals, human escalations. Color-coded nodes: success/warning/failure/escalated.
  • Replay system: load any past execution, step through decisions, modify input at any step, compare diverged execution against original. Invaluable for diagnosing failures.
05End-to-End TestingDays 12–13
  • Demo scenario: research task requiring web search, data extraction, analysis, written summary. Show: supervisor planning, parallel specialist execution, reviewer catching an issue, human approving final deliverable.
  • Containerize: FastAPI + Redis + PostgreSQL + Celery workers + trace explorer UI + human review UI. Include demo script that runs the showcase automatically.
06Polish for PortfolioDays 13–14
  • Frame: "I built a multi-agent orchestration system where AI agents decompose complex tasks, use tools via MCP, learn from past interactions via persistent memory, and escalate to humans when confidence is low. This is production infrastructure for autonomous AI workflows — not an AI demo."
