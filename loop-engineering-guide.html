THE EVOLUTION

AI coding didn't stay still. In just a few years, the leverage point shifted four times. Each step made the previous one look primitive. Here's what actually changed.

Prompt Engineering

You crafted clever prompts to get better answers from AI. Write better → get better output. One shot, one result. The human was doing all the thinking about how to ask.

Context Engineering

You realised what goes around the prompt matters more than the prompt itself. Docs, examples, system instructions, memory. Fill the context window with the right information and the model surprises you.

Harness Engineering

You stopped running AI manually and put it inside a framework. Claude Code, Codex, Cursor. The harness handles memory, tools, file access, and agent coordination. You write rules, not prompts.

→ Loop Engineering

You stop prompting entirely. You design the system that prompts the agent, runs it on a schedule, verifies the output, and loops again until the goal is complete. You are no longer in the loop. The loop runs without you.

"I don't prompt Claude anymore. I have loops running that prompt Claude and figure out what to do. My job is to write loops."

— Boris Cherny, Head of Claude Code at Anthropic

"You shouldn't be prompting coding agents anymore. You should be designing loops that prompt your agents."

— Peter Steinberger
Section 02 — Beginner

WHAT IS A LOOP?

Think of it like a night-shift employee who doesn't need managing. You give them a goal before you leave. They work through the night, check their own work, fix their mistakes, and leave a report on your desk by morning.

A loop is a small program that runs on a schedule with agent intelligence built in. Instead of typing a prompt and waiting for a reply, you define an outcome — and the loop figures out the steps, runs them, checks the result, and repeats until done.

The Old Way vs The Loop Way

Old way
  • → You open ChatGPT
  • → You write a prompt
  • → You check the output
  • → You fix errors
  • → You send it back
  • → You are the loop
Loop way
  • → You define the goal
  • → The loop finds the work
  • → AI generates output
  • → AI verifies output
  • → Loop runs again
  • → You get a report
Key difference
With ChatGPT, you remove yourself from the loop when you're satisfied. With loop engineering, you were never in the loop to begin with. The AI generates, the AI verifies, and it keeps running until the output is correct.

THE 5 PIECES OF EVERY LOOP

Every powerful loop — no matter what tool you use — has these five components. Miss one and the loop breaks down.

1
Automation
Finds the work

The loop runs on a schedule — every hour, every day, every 15 minutes. It doesn't wait for you to start it. It wakes up, finds what needs doing, and begins.

Example: Every morning at 8am, the loop scans your GitHub issues for anything new, labelled or unassigned, and queues them for the agent.
2
Parallel Agents
Work simultaneously

Multiple agents work at the same time in isolated environments (called worktrees) so they can't break each other. One writes, one checks. Both run at once.

Example: While Agent A is fixing a bug, Agent B is already writing the test for it — in a separate branch that can't conflict.
3
Skills / Context
Agents know your project

The agent already has context about your codebase, your style, your conventions — loaded before it starts. You don't re-explain your project every time.

Example: A CLAUDE.md file in your project tells the agent which patterns to use, which files to touch, and what "done" looks like.
4
Connectors
Agents plug into real tools

The agent isn't just writing code — it's opening GitHub PRs, posting Slack updates, updating your CRM, sending emails. It reaches into your actual workflow.

Example: After fixing a bug, the loop opens a PR, assigns it, and posts a summary in your #dev Slack channel — without you touching anything.
5
Subagents
One writes, one checks

The maker/checker split. One subagent generates the output. A completely separate subagent reviews it with fresh eyes. The loop only exits when the checker approves.

Example: Agent A writes the code change. Agent B runs the tests, checks the diff, and either approves or sends it back with notes. Human sees only the approved result.
The result
You're not in the middle checking outputs anymore. The AI generates, the AI verifies, and it keeps going until the goal is met. You get a report.
Section 04 — Intermediate
04

LOOPS RUNNING LOOPS

One loop can start other loops — controlling entire agent teams while you sleep. This is where things get powerful.

How It Works

Think of it as a company hierarchy — but made of AI. You have an orchestrator loop at the top that routes work to specialist loops below it. Each specialist loop is optimised for exactly one job.

🎯
Orchestrator Loop
Runs daily. Checks what work exists. Decides which specialist loop to trigger. Verifies final outputs. Reports to you.
↓ triggers
🐛
Bug Fix Loop
Scans issues, fixes bugs, opens PRs.
🧪
Test Loop
Monitors CI failures, writes missing tests.
📦
Dependency Loop
Updates packages, checks compatibility.
📝
Changelog Loop
Drafts release notes from git history.
↓ results flow back up
📋
You — Morning Report
PRs opened, bugs fixed, tests written. Here's what needs your decision.

Real Use Cases

Content creator
Researcher loop scrapes competitors → CMO loop writes scripts in your voice → Publisher loop posts and tracks → Analyst loop reports what worked → next loop scripts more of that.
Developer
Triage loop categorises issues → Bug loop fixes P1s → PR Babysitter loop watches CI → Changelog loop drafts release notes. All without you touching it.
Business owner
Lead research loop scores inbound leads → Sales prep loop writes briefing docs → Follow-up loop drafts emails → Analytics loop tracks what closed. You focus on the call.

"In 12 months, this will be the difference between businesses running on compounding systems that are impossible to catch — and businesses that aren't."

— Shaun Pervis

Start in 5 Minutes

The easiest way to get started is the loop-engineering GitHub repo — it has 7 production-ready patterns you can run immediately with no clone required.

GitHub Repo
github.com/cobusgreyling/loop-engineering  · ⭐ 3,400+ stars · MIT License

Step 1 — Scaffold your first loop (Daily Triage pattern):

bash
npx @cobusgreyling/loop-init . --pattern daily-triage --tool claude-code

Step 2 — Estimate token cost before you run:

bash
npx @cobusgreyling/loop-cost --pattern daily-triage --level L1

Step 3 — Audit your loop readiness score:

bash
npx @cobusgreyling/loop-audit . --suggest

Step 4 — Start in report-only mode first (safe):

Claude Code
/loop 1d Run loop-triage. Update STATE.md. No auto-fix in week one.

The 7 Production Patterns

Daily Triage — categorise issues every day Low cost
PR Babysitter — watch CI every 15 minutes High cost
CI Sweeper — fix failing tests automatically Very high cost
Dependency Sweeper — update packages daily Medium cost
Changelog Drafter — write release notes on tag Low cost
Post-Merge Cleanup — tidy up after merges Low cost
Issue Triage — label and sort new issues Low cost
Rule of thumb
Start at L1 (report only) — the loop tells you what it would do, but doesn't do it. Once you trust it, move to L2 (assisted), then L3 (unattended). Never skip to L3 on day one.
